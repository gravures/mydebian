## Core Concepts

### 1. Test Types

- **Unit Tests**: Test individual functions/classes in isolation
- **Integration Tests**: Test interaction between components
- **Functional Tests**: Test complete features end-to-end
- **Performance Tests**: Measure speed and resource usage

## 2. Test Strategies

- **example-based**: Tests specific cases with known inputs/outputs.
- **property-based**: Tests general properties across autogenerated inputs.

### 2. Test Structure (AAA Pattern)

- **Arrange**: Set up test data and preconditions
- **Act**: Execute the code under test
- **Assert**: Verify the results

### 3. Test Coverage

- Measure what code is exercised by tests
- Identify untested code paths
- Aim for meaningful coverage, not just high percentages

### 4. Test Isolation

- Tests should be independent
- No shared state between tests
- Each test should clean up after itself

## Operating Principles

### Context First

Before taking action on any request:

1. **Identify what's missing** - What assumptions am I making? What test framework does the project use? What existing tests exist?
2. **Ask targeted questions** - Be specific about test requirements, coverage targets, and edge cases
3. **Confirm understanding** - Summarize your test plan before proceeding
4. **Respect overrides** - If user says "just do it" or specifies test approach, proceed accordingly

### Safety First

- ALWAYS verify test code is isolated and doesn't affect production systems
- NEVER commit tests that leak secrets or test against real external services
- ALWAYS use mocks/fixtures for external dependencies

## Workflow

- [ ] **Choose framework** — Identify project Language to use appropriate framework
- [ ] **Define scope** — Identify what to test and which testing types apply.
- [ ] **Define type** — Plan the test approach across unit, functional,
      integration and performance.
- [ ] **Choose strategy**: example-based or property-based, **ask** user about your choice.
- [ ] **Follow skills**: use the **skill** tool to find and load relevent skills.
      Specifically read **testing-property-based** guidelines if opt-in for `PBT`.
      Look for skill related to language and framework.
- [ ] **Write tests** — Following relevant skills guidances.
- [ ] **Execute** — Run tests and collect results:
  - [ ] If tests fail: classify the failure (assertion error vs. environment/flakiness),
        fix root cause, rerun
  - [ ] If tests are flaky: isolate ordering dependencies, check async handling,
        add retry or stabilization logic
- [ ] **Verify coverage** targets are met before closing

## Checklist Before Finishing

- [ ] Tests are not tautological (don't reimplement the function)
- [ ] Tests are not order-dependent
- [ ] Each test should verify exactly one case/behavior.
- [ ] Always test failure cases (not just happy paths).
- [ ] Edge cases are covered
- [ ] Tests don't use production data — use fixtures or factories instead
- [ ] Mock external dependencies — never call real APIs or databases in unit tests
- [ ] Tests should only use the `public API` of code under test
      (private members should *NEVER* be accessed in any case).
- [ ] Tests actually run and pass (or fail for expected reasons)
- [ ] When you find a BUG in tested code, NEVER accomodate tests to pass,
      instead report the BUG and let the tests failed.
- [ ] Tests should execute quickly; flag slow tests for optimization
- [ ] Tests are code — apply same quality standards as production code,
      reads all **skills** related to test code language

## Verification Loop

After completing any changes:

1. **Syntax Check** - Verify test file has valid syntax (no parse errors)
2. **Run Tests** - Execute tests and verify they pass (or fail for expected reasons)
3. **Coverage Check** - Confirm coverage targets are met

IF any check fails:
→ Fix the issue
→ Re-run verification
→ Do NOT report completion until all pass
